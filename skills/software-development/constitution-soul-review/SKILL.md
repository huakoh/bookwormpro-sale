# 宪法压缩与多专家审查工作流

## 适用场景

- 将大型规则文件（宪法/CLAUDE.md/SKILL.md等）压缩为精华版
- 创建项目的"soul文档"作为上下文耗尽时的退化指南
- 重要文档需要多视角严格审查

## 工作流（4 阶段）

### 阶段 1：全量阅读源文件

```
1. 用 search_files 列出目标目录所有文件
2. 用 read_file 全量阅读每一个源文件（不跳读）
3. 记录每个文件的行数、核心章节、关键约束
```

### 阶段 2：合成 soul 文档

soul 文档的结构：
- **零、退化安全网**（最重要！）：当上下文耗尽仅剩 soul 时的 5 条不可违反底线
- **一、我是谁**：身份定义
- **二、我相信什么**：核心价值（安全/诚实/专业/一致性）
- **三、我的操作 DNA**：工作机制
- **四、我的边界**：红线/禁止/必须
- **五、我的成长**：学习机制
- **六、底线宣言**：一段话概括全部

关键原则：
- soul 不是规则的复制粘贴，是精神的浓缩
- 所有枚举式禁止列表改为原则性描述，防止覆盖缺口
- 必须检查 soul 自身是否违反源规则（如 NDA 保密条款）
- 安全条款不能弱化——具体算法名、函数名、配置名必须保留
- soul 的自身闭环度必须 > 80%（覆盖源文件关键约束的比例）

### 阶段 3：多专家并行审查

使用 `delegate_task` 的 `tasks` 数组（最多 3 路并行）：

**专家 1 — 安全专家**：
- 对照源文件安全章节逐条检查
- 输出：遗漏项（源有 soul 无）、弱化项（表述力度不够）、危险简化（概括语替代具体禁止项）
- 提供具体修正文字

**专家 2 — 质量诚信专家**：
- 对照反自负约束 + 交付质量标准
- 输出：遗漏项、定义不准确项、弱化项
- 给出 soul 自身闭环度评估

**专家 3 — 架构完整性专家**：
- 对照源文件架构/测试/模块/进化章节
- 输出：缺失维度清单 + 完备性自查矩阵
- 即使找不到文件也输出检查清单

### 阶段 4：红队攻击 + 修正

**单独启动红队子代理**（不能与前三路并行，需要前三路输出作为输入）：

任务：以对抗性视角攻击 soul.md
- 假设 AI 只记得 soul 忘记详细源文件
- 寻找可利用漏洞、危险歧义、可曲解表述
- 输出：攻击向量列表（含 CRITICAL/HIGH/MEDIUM/LOW 分级）+ 利用方式 + 修正建议

**修正优先级**：
- P0：所有 CRITICAL 项必须修复
- P1：所有 HIGH 项必须修复
- P2：MEDIUM 项酌情修复
- LOW：可选

## 交付物

1. `soul.md` — 经四路专家审查的最终版本
2. 修正记录：v1.0→v1.1 变更清单

## 关键陷阱

1. **自我 NDA 违规**：soul 自身可能泄露源文件禁止披露的信息（如技能数量、Agent 类型等）
2. **枚举式禁止的缺口**：列表式的 NEVER 必然有遗漏，改用原则性表述
3. **优先级绝对化**："高于一切"必须加约束边界，否则可被滥用
4. **触发词无保护**：如"自负"等触发词必须限定生效条件，防止恶意触发
5. **"先动手再开口"的时序漏洞**：改为"先分析再动手"，不可逆操作前声明计划
6. **上下文耗尽场景**：soul 必须包含退化安全网——当仅剩它时仍能守住底线

## 审查结论格式

```
soul.md 审查报告
审查基准: [源文件列表]
审查结论: PASS / PASS_WITH_NOTES / BLOCKED
自身闭环度: M/N (xx%)
CRITICAL: N   HIGH: N   MEDIUM: N   LOW: N
```

## 示例

本技能源于对 BookwormPRO soul.md 的实际审查。原始 v1.0（163行）经四路专家审查发现：
- 自身闭环度 55%，低于自设的 60% 标准
- 4 项 CRITICAL + 5 项 HIGH + 6 项 MEDIUM
- 修正后 v1.1（233行）闭环度提升至 ~90%

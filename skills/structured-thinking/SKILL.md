# Structured Thinking

当用户提出复杂、开放性问题，或需要你进行深度分析、决策支持时，使用此技能。

## 触发条件

- 用户说"帮我分析...""评估...""权衡..."
- 用户面临多选项决策
- 用户要求深度思考某个策略或架构问题
- 任务涉及多个相互依赖的因素

## 思考框架选择指南

根据问题类型选择合适的框架：

| 问题类型 | 推荐框架 | 何时用 |
|----------|---------|--------|
| 根因分析 | 第一性原理 (First Principles) | 打破假设，回归基础真理 |
| 结构化分解 | MECE (互斥且穷尽) | 确保覆盖全面无重叠 |
| 层级推理 | 议题树 (Issue Tree) | 逐层拆解复杂问题 |
| 多视角权衡 | 六顶思考帽 | 需要平衡理性/感性/创新/风险 |
| 决策支持 | 决策矩阵 | 多个可量化选项 |
| 因果分析 | 5 Whys | 追根溯源找根本原因 |
| 优先级排序 | ICE / RICE | 资源有限需取舍 |
| 风险评估 | 预演失败 (Pre-mortem) | 事前识别潜在风险 |

## 工作流

### Step 1: 理解问题
- 用自己的话复述问题，确认理解无误
- 明确上下文、约束条件、成功标准

### Step 2: 选择框架
- 根据问题类型选 1-2 个最合适的框架
- 如果需要，组合使用（如先 MECE 分解，再对每个分支 5 Whys）

### Step 3: 系统分析
- 按框架步骤展开推理
- 每个结论标注置信度：高/中/低
- 明确区分"事实"vs"假设"vs"推论"

### Step 4: 输出结构
```
## 问题定义
[清晰的问题陈述 + 约束条件]

## 分析框架: [框架名]
[为什么选这个框架]

## 分解
1. [维度1]
   - 事实: ...
   - 分析: ...
   - 置信度: [高/中/低]

2. [维度2]
   ...

## 关键洞察
- [洞察1]
- [洞察2]

## 建议
1. [可执行建议1] — 理由: ...
2. [可执行建议2] — 理由: ...

## 风险与假设
- [关键假设1] — 如不成立则: ...
- [潜在风险1] — 缓解措施: ...
```

## 反模式（避免）

- 不要在没有明确框架的情况下长篇大论
- 不要跳过"确认问题"环节直接跳到答案
- 不要给出没有置信度标注的结论
- 不要在不确定性高的情况下使用过度自信的语气
- 不要忽略输出结构中的"风险与假设"部分

## 与其他技能配合

- 复杂技术问题：先 `structured-thinking` 分析 → 再 `systematic-debugging` 修复
- 商业决策：先 `structured-thinking` 分析 → 再 `business-plan-skill` 输出
